URI和URL

  • URI
  • URL
network

一直以来对于URL和URI这两个概念的区别和联系都有一定的困惑，今天又遇到这两个概念，本着刨根问底的精神搜了下相关资料终于把这两个概念弄清楚了，感谢万能的google。

1、URI，是uniform resource identifier，统一资源标识符，用来唯一的标识一个资源。URI是一个抽象的高层次的概念。一个URI实例可以是相对的也可以是绝对的，此外URI还会有一定的语法规则。 2、URL，是uniforresourcelocator，统一资源定位器，它是一种具体的URI，符合URI的语法规则，即URL可以用来标识一个资源，而且还指明了如何定位这个资源，所以URL不能是相对的，schema必须被指定。

举个例子：从HttpServletRequest的javadoc中可以看出，getRequestURI返回一个String，“the part of this request’s URL from the protocol name up to the query string in the first line of the HTTP request”，比如“POST /some/path.html?a=b HTTP/1.1”，则返回的值为”/some/path.html”。现在可以明白为什么是getRequestURI而不是getRequestURL了，因为此处返回的是相对的路径。而getRequestURL返回一个StringBuffer，“The returned URL contains a protocol, server name, port number, and server path, but it does not include query string parameters.”，完整的请求资源路径，不包括querystring。

最后总结一下，URI仅仅是唯一标识的字符串，没有别的含义了，所以实现方式比较自由，可以是绝对的也可以是相对的,具体实现呈现出来的样子可以跟资源的路径一定关系也没有。而URL不仅唯一标识一个资源而且还包含了定位该资源的信息，URL每部分都有具体含义，所以不能是相对的。

ps： java.net.URL类不提供对标准RFC2396规定的特殊字符的转义，因此需要调用者自己对URL各组成部分进行encode。而java.net.URI则会提供转义功能。因此The recommended way to manage the encoding and decoding of URLs is to use java.net.URI. 可以使用URI.toURL()和URL.toURI()方法来对两个类型的对象互相转换。对于HTML FORM的url encode/decode可以使用java.net.URLEncoder和java.net.URLDecoder来完成，但是对URL对象不适用。
